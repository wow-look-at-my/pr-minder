# pr-minder

Cloudflare Worker (TypeScript) that acts as a GitHub App webhook handler for keeping PRs up to date.

## Docs serving

`worker.ts` serves public docs on GET/HEAD: `/` and `/llms.txt`. `llms.txt` is the single source of truth; `index.html` is a self-contained page that fetches `/llms.txt` and renders the Markdown. Both are **gzipped at build time** (`scripts/build-docs.mjs`, wired via `wrangler.toml [build].command`, output git-ignored) and imported as `*.gz` blobs (the `[[rules]] type = "Data"` makes them `ArrayBuffer`). Serving negotiates on `Accept-Encoding` (and sets `Vary: Accept-Encoding`): clients that accept gzip get the blob as-is ‚Äî `new Response(gz, { encodeBody: "manual", headers: { "content-encoding": "gzip" } })`, where `encodeBody: "manual"` is required else the runtime recompresses the already-gzipped bytes; clients that don't get the blob decompressed server-side via `DecompressionStream` and served identity. Keep doc content in `docs/*` (the "no embedded docs in source files" rule); never inline it in a `.ts`. Install URL: `https://github.com/apps/pr-minder/installations/new`. Webhooks are POST, so they never collide with the GET docs routes. `verifyWebhook` lives in `webhook.ts` so the test suite doesn't pull `worker.ts`'s binary imports through vite.

## Build / typecheck

`tsc --noEmit` (type-check only) is wired into **both** the build and the tests, so a type error
can't slip through either path. This matters because esbuild ‚Äî what vitest and wrangler bundle with ‚Äî
strips types **without checking them**, so `tsc` is the only thing standing between a type error and a
green test run / a production deploy. Do NOT run `tsc` or `wrangler` directly. Use:
```sh
npm run typecheck   # tsc --noEmit (standalone)
npm test            # tsc --noEmit && vitest run     (tests can't pass with a type error)
npm run build       # tsc --noEmit && npm run build:docs
npm run dev         # wrangler dev ‚Äî runs `npm run build` (so typecheck) before bundling
```

`wrangler.toml`'s `[build].command = "npm run build"`, so `dev`, `deploy`, and Cloudflare Workers
Builds all typecheck first and a non-zero `tsc` exit aborts the build/deploy. There is no Go in this
project ‚Äî do not use `go-toolchain`.

## Deployment ‚Äî NEVER deploy manually

It is **never** valid to deploy by hand. Do NOT run `npm run deploy`,
`wrangler deploy`, or any other deploy command ‚Äî not even to "test" or "ship a
fix faster". Cloudflare's Git integration watches this repo and deploys
automatically: a preview Worker for every PR branch, and **production on merge to
`master`** (the repo's default branch). Manual deploys bypass that pipeline and are
never the right move ‚Äî ship by merging the PR.

## GitHub API base (github-state-mirror)

`github.ts` routes **every** GitHub API call through `API_BASE` instead of hardcoding `https://api.github.com` ‚Äî reads, REST writes, the App-level JWT endpoints, and the auto-merge GraphQL mutations alike. `configureApi(base, appId, privateKey)` (called from both `worker.ts` entry points, `fetch` and `scheduled`) sets it from the `GITHUB_API_BASE` var (wrangler.toml `[vars]` = `https://github-state-mirror.pazer.io`); absent the var it defaults to `https://api.github.com`, so dev/tests are unchanged. The mirror is a **drop-in GitHub API proxy** (`wow-look-at-my/github-state-mirror`): it serves a small set of webhook-fed cached routes whose responses are **byte-for-byte identical to GitHub's** (currently just the org-repos `POST /graphql` read) and transparently forwards everything else to GitHub uncached, so pointing the base at it is safe even for endpoints it doesn't cache.

**Identity / cache partition.** The mirror partitions its cache per caller. pr-minder's install tokens rotate hourly, which would give a fresh empty bucket each hour ‚Äî so on every install-token request (`gh()` and `graphql()`) `mirrorIdentity()` attaches the App JWT (minted via `appJWT`, cached per isolate, re-minted ~1 min before its ~9-min expiry) as the **`x-mirror-identity`** header. (The App-JWT-authed endpoints below skip it ‚Äî they're passthrough and already carry the App JWT as `Authorization`.) The mirror verifies it (`GET /app`) and partitions us as `app:<id>` ‚Äî one stable, webhook-warm bucket for all our rotating tokens. The `Authorization: Bearer <install-token>` header is still sent and is what the mirror uses for on-demand upstream fetches, so per-repo authorization is preserved. If app creds aren't configured the header is omitted and the mirror falls back to token-fingerprint partitioning (still correct, just not shared).

**Nothing stays on api.github.com directly.** Every call in `github.ts` ‚Äî the `gh()` reads, the REST writes (`updateBranch`/`createPull`/`addLabelsToPr`/`ensureLabel`/`closePull`/`commentOnPr`/`updatePullTitle`/`mergePullRequest`/`removeLabelFromPr`/`deleteBranch`/`setPullState`), the App-JWT endpoints (`installToken`, `listInstallations`, `repoInstallationId`, `appBotLogin`), the auto-merge GraphQL mutations (`enableAutoMerge`/`disableAutoMerge` via `graphql()`), `compareCommits`, and the `getPullDiff` files-API fallback (`assembleDiffFromFiles`) ‚Äî goes through `API_BASE`. The old `ghDirect()` cache-bypass is **gone**: the mirror no longer caches `/compare` or `/pulls/{n}/files` with trimmed shapes (it passes them through verbatim now), so there is nothing to bypass ‚Äî `compareCommits`'s `files`/`changed_files` (the empty-PR gate) and the diff fallback's per-file `patch` come back intact through the proxy. A **guard test** in `github.test.ts` asserts the source contains no ``fetch(`https://api.github.com`` literal, so a direct call can't regress back in (the `API_BASE` default constant, `resetApiConfig`, and the `usingMirror` comparison are config, not `fetch(` calls ‚Äî they stay).

How the mirror handles each (identical-or-passthrough ‚Äî see its CLAUDE.md):

- **Cached, identical:** the org-repos `POST /graphql` read ‚Äî assembled to match GitHub's selection set exactly, locked by an identity test on the mirror side.
- **Passthrough (reverse-proxied verbatim, uncached):** everything else ‚Äî the App-JWT endpoints (the mirror can't validate a JWT as a partition identity, but its proxy forwards the `Authorization` JWT untouched), the non-org GraphQL mutations (the `/graphql` handler forwards any non-org query to GitHub with the install token, and `requireAuth` accepts it via the `x-mirror-identity` header), `compareCommits`, `pulls/{n}/files`, the single-PR/list reads (`getPull`, `listOpenPulls`, `searchPrsByLabel`, branches, reviews, `actions/runs`), and the REST writes.

To revert the migration, set `GITHUB_API_BASE = "https://api.github.com"` (or remove the var) ‚Äî `API_BASE` falls back to GitHub directly and the `x-mirror-identity` header becomes a harmless no-op (GitHub ignores the unknown header).

## Config file loading order

1. `{repo}/.github/config/pr-minder/pr-minder.jsonc` ‚Äî per-repo (highest priority)
2. `{org}/.github` repo, `.github/config/pr-minder/pr-minder.jsonc` (same path) ‚Üí `repos.{repo}` field for per-repo overrides, top-level for org defaults
3. No config found ‚Üí everything disabled (opt-in design; nothing fires by default)

`loadConfig` runs on essentially every webhook event and each resolution is up to two Contents API calls (per-repo file, then the org `.github` file ‚Äî the common case is a 404 + a hit/404). To avoid hammering the API for config that rarely changes, the resolved result is **memoized per `owner/repo` for 60s** (`CONFIG_CACHE_TTL_MS`) in a module-scope `Map` ‚Äî same per-isolate-lifetime pattern as `handlers.ts`'s `labelCheckedAt`. Consequences: a config edit takes up to 60s to take effect, and the cache key is `owner/repo` only (the resolved config is identical regardless of which install token reads it). Only **definitive** resolutions are cached: a transient fetch failure (5xx/throttling/network ‚Äî `fetchRepoFile` throws on any non-2xx that isn't a 404) degrades to "disabled" for that one event but is **not** cached, so the next event retries instead of being stuck with an empty config for a full TTL. A 404 (file absent) and a malformed-JSON file are definitive and *are* cached.

Separately, the **org `.github` file is memoized per `owner`** (`orgConfigCache`, also 60s) inside `loadOrgConfig`, because that one file is shared by every repo in the org. The `owner/repo` cache above only de-dupes repeated reads of the *same* repo; it does nothing for a pass over many *different* repos in one org. Without the owner-level cache, a cross-owner sweep re-fetched the shared org file once per repo ‚Äî the dominant cost that, at ~380 repos, blew Cloudflare's 50-external-subrequest cap in a single invocation. Same cacheability rule: a 404/malformed org file is cached (as "no org config"), a transient fetch failure is not. `loadOwnerConfig(owner)` returns just the org-level config (no per-repo file or override) and is what the auto-merge backstop uses to learn the `auto_merge` label names for its search.

## Config shape

`auto_update_pr.triggers`: approval-gate conditions; any one passing fires `update-branch` (skipped when the merge would be a no-op ‚Äî see "No empty merge commits" under Key invariants). Keys within a condition are ANDed; conditions are ORed. Supported keys: `label`, `approved_by`, `min_approvals`.

`auto_label_pr`: map of label name ‚Üí `{ auto_add, create_label_if_missing_in_repo, color, mode, auto_merge_method }`.

- `auto_add: "on_pr_creation"` ‚Äî apply label when a PR is opened.
- `create_label_if_missing_in_repo: true` ‚Äî create the label in the repo (with `color`) if absent.
- `mode: "auto_update"` ‚Äî when this label is present on a PR, keep the branch in sync with its base (complementary to `auto_update_pr.triggers`).
- `mode: "auto_merge"` ‚Äî bidirectionally syncs this label with GitHub's native auto-merge. Adding the label enables auto-merge; removing it disables auto-merge. `auto_merge_enabled`/`auto_merge_disabled` webhook events add/remove the label. **Whenever GitHub refuses to *arm* native auto-merge** (`enablePullRequestAutoMerge` returns HTTP 200 + an `errors[]` array), `enableAutoMerge` falls back to merging the PR directly via `mergePullRequest` (REST `PUT .../merge`) with `auto_merge_method`. The two cases that hit this: the PR is **already mergeable** (`"clean status"` ‚Äî nothing pending for auto-merge to wait on) and the repo has **not enabled "Allow auto-merge"** in its settings (`"Auto merge is not allowed for this repository"` ‚Äî pr-minder has no `administration` permission, so it can't toggle that setting itself). The fallback is unconditional on logical-error responses precisely so the label means "merge this" even in repos without native auto-merge enabled; `mergePullRequest` is the final authority, so branch protection still gates the merge and a not-yet-mergeable PR comes back as a swallowed 4xx (never merged). When native auto-merge *can* be armed, the success path returns first, so pr-minder never pre-empts GitHub's wait-for-checks behavior. (Limitation: in a repo without native auto-merge, a labeled PR with pending required checks won't be merged later automatically ‚Äî pr-minder isn't subscribed to check/status completion events ‚Äî so enable "Allow auto-merge" on the repo if you want the wait-for-checks behavior.)
- `mode: "merge_conflict"` ‚Äî pr-minder owns this label: it's **added when the PR has a merge conflict** (`mergeable === false`) and **removed when the conflict clears** (`mergeable === true`); it's never set by hand. Because GitHub computes `mergeable` asynchronously and emits no webhook with the result, the flow is enqueue-then-settle: a PR's own events (`opened`/`reopened`/`synchronize`, via `evaluateMergeConflict` in `onPR`) read it fresh and act when known, else drop a `conflict:` KV reminder; a default-branch push (`onPushToDefault`) and the install/first-webhook backfill flag *every* open non-draft PR (cheap KV puts, no GitHub calls), since any of them may now conflict. The cron's `runConflictChecks` drains the reminders (one `getPull` per pending PR, budget-bounded), applying the label once `mergeable` is non-null and clearing the reminder; a still-`null`/transient one is left for the next tick (TTL ~1 day bounds it), a closed/draft/gone or feature-off one is cleared. `mergeable` reflects *conflicts only* (not checks/reviews ‚Äî those live in `mergeable_state`), so `false` is exactly a merge conflict. All entry points gate on `conflictLabelNames(config).length > 0`, so the feature costs nothing where it isn't configured. Every label op is idempotent (add-if-missing / remove-if-present), so re-evaluating a settled PR is a no-op.
- `auto_merge_method` ‚Äî `"merge"`, `"squash"` (default), or `"rebase"`. Only used when `mode` is `"auto_merge"`.

`auto_trigger_workflows`: boolean (default false). Revives "zombie" PRs ‚Äî author `github-actions[bot]` (created with the default `GITHUB_TOKEN`, whose events GitHub won't let trigger workflows) and no workflow runs for their head commit ‚Äî by closing+reopening with the App token (`retriggerWorkflows` in `github.ts`), which fires a fresh event that DOES run their CI. Gated on `auto_trigger_workflows` + bot-author throughout.

One core check, `reviveIfZombie(env, repo, pr, ‚Ä¶)`, is shared by every trigger: bot-author-gated, it spends a `hasWorkflowRuns` call (`GET /actions/runs?head_sha=‚Ä¶`) and close+reopens if there are none. It is **check-once via Workers KV** (`state.ts`): keyed `pr:{repo}#{num}` ‚Üí the head SHA last evaluated; if the current SHA matches it's skipped, else it's evaluated and the SHA recorded. A new commit changes the SHA, so a *touched* PR is re-checked while an untouched one never is. **The first commit we ever handle for a PR (no prior SHA recorded) is revived immediately** ‚Äî a bot PR is born with zero CI, so that reading is trustworthy. **A *follow-up* commit with zero runs is only revived once it has aged past `REVIVE_MIN_AGE_S` (~60s, in `handlers.ts`)**, fetched via `commitAgeSeconds` (`GET /repos/{repo}/commits/{sha}`); a still-fresh one is left *unrecorded* and a `recheck:{repo}#{num}` reminder is dropped in KV (`setRecheck`), because webhooks won't re-fire on their own once the commit ages. This is what stops pr-minder's **own `update-branch` merge** ‚Äî a new head SHA whose runs haven't registered yet ‚Äî from being close+reopened a second time, without needing to know who pushed it. (On a commit-age lookup error `commitAgeSeconds` returns null and the follow-up is deferred, failing safe against a spurious re-close.) The reminder is drained by `runRechecks` (next bullet); on a verdict (revived, or runs found) the reminder is cleared.

Triggers (event-driven, plus one narrow cron for deferred re-checks):
- Live `pull_request` **opened / reopened / synchronize** ‚Üí `reviveIfZombie` on that PR, gated by `shouldConsiderRevive(action, sender)` (all three actions qualify; the only one dropped is a `reopened` whose `sender.type === 'Bot'` ‚Äî our own close+reopen coming back ‚Äî so the cycle can't loop). `synchronize` is included so new commits get re-checked; the **commit-age guard inside `reviveIfZombie`** (not the event sender) is what keeps pr-minder's own `update-branch` merge from being closed+reopened a *second* time ‚Äî that merge is a follow-up commit still too fresh to have registered CI, so it's deferred. If it reopened, `onPR` returns and the fresh event drives the rest.
- **Cron, every 5 min** (the Worker's `scheduled` handler) ‚Üí two cheap passes: (1) `runRechecks` drains the `recheck:` reminders left by deferred follow-up commits, re-evaluating each PR once it has aged. This is **not a poll over all PRs**: it reads only the reminders (`listRechecks`, a KV list), so with none pending it makes a single KV call and **zero GitHub API calls**. Per pending PR it mints a token for that repo (`repoInstallationId` ‚Üí `installToken`, cached per run), refetches the PR (`getPull`), and re-runs `reviveIfZombie`. Closed/merged PRs clear their reminder; a missing token or transient fetch failure leaves it for the next sweep (KV TTL ~1 day bounds an abandoned one). (2) `reconcileAllInstalls` runs the auto-merge backstop across every installation (see below).
- **First webhook from a repo** ‚Üí `maybeBackfillRepo` (run from the opportunistic per-event block in `handle()`): the event-driven "check at least once" for repos installed before a feature existed (GitHub never re-sends their install event). It records, per repo, *which* catch-up capabilities have been swept (`backfill:{repo}` = a CSV subset of `BACKFILL_CAPS`: `zombie`/`openpr`/`conflict`/`describe`), not a single boolean ‚Äî so a feature **enabled after the first sweep** (e.g. turning on `auto_open_pr` later) is noticed (its capability is absent from the set) and that catch-up re-runs. Each cap maps to one sweep: `zombie`‚Üí`maybeRetriggerZombiesForRepo`, `openpr`‚Üí`maybeOpenPrsForRepo`+`closeEmptyAutoPrs`, `conflict`‚Üí`enqueueConflictChecksForRepo`, `describe`‚Üí`enqueueDescribeChecksForRepo`. Cheap by construction: once every capability is recorded it's a single KV read (the permanent gate); while incomplete, the re-check that needs config (a token mint + a memoized Contents read) is throttled per isolate (~15 min, like `labelCheckedAt`), so the initial backfill still runs promptly but later events don't reload config every time. A **legacy ISO-timestamp value** (from the old boolean flag) reads as the empty set, so existing repos re-sweep once under their current config on the next event after deploy. (This was the fix for `auto_open_pr` being enabled on an already-installed repo and never opening PRs for its pre-existing branches ‚Äî the one-time flag had already been set before the feature was turned on.)
- `installation`/`installation_repositories` ‚Üí the same `maybeRetriggerZombiesForRepo` sweep for the install's repos (plus the other sweeps), then records the swept capability set (`enabledBackfillCaps(config)`).

Auto-merge backstop (`reconcileInstall` / `reconcileAllInstalls`): the safety net for an `auto_merge`-labeled PR the live webhook path dropped (e.g. a webhook whose invocation hit the subrequest cap ‚Äî the original failure mode that left PRs silently unmerged, no error, HTTP 200). `reconcileInstall(owner, token, log, budget)` loads the **owner-level** config (`loadOwnerConfig`) for the `auto_merge` label names, then for each label **searches the whole installation** (`searchPrsByLabel` ‚Üí `GET /search/issues?q=is:pr is:open draft:false label:"‚Ä¶"`; an install token scopes results to the install's repos, so no `org:` qualifier) and on each hit not already armed calls `enableAutoMerge` (which direct-merges a PR that's already mergeable, so a "ready" PR gets merged instead of sitting forever). **Cost scales with *labeled* PRs, not repo count** ‚Äî one search covers an owner's entire fleet (a 500-PR repo with 3 labeled PRs returns 3), which is what makes this safe to run fleet-wide in a single invocation. Per-PR work needs a `getPull` (to read `node_id`/`auto_merge`) and is wrapped in try/catch; `budget.calls` bounds the GitHub calls so an invocation stays well under the 50-external-subrequest cap, with anything not reached picked up next time.

It runs in three places: (1) **deploy** ‚Äî `startupReconcile(env)` calls `reconcileAllInstalls` (iterate installs ‚Üí token ‚Üí `reconcileInstall`), still **gated once per deploy** by the version KV flag (`startup:{CF_VERSION_METADATA.id}`), claimed *before* sweeping so a concurrent isolate bails; fired from `worker.ts` on the first request via `ctx.waitUntil` + a per-isolate module flag. (2) **cron, every 5 min** ‚Äî `reconcileAllInstalls` again (a fresh invocation, larger budget). (3) **every webhook** ‚Äî after `dispatch`, `handle()` runs `reconcileInstall` for the webhook's owner, **cooldown-gated per owner** via a `swept:{owner}` KV marker (`recentlySwept`/`markSwept`, ~60s TTL) so a burst of webhooks can't exceed GitHub's ~30/min search rate limit; the marker is claimed before running so concurrent isolates don't pile on, and failures are swallowed (the backstop must never fail the webhook). This is self-feeding: a merge here emits its own push webhook, which drives the next pass. Steady state is still driven by the live `labeled`/`unlabeled`/`auto_merge_enabled`/`auto_merge_disabled` webhooks (immediate); the backstop only catches what they dropped.

The older per-repo `reconcileAutoMerge(repo, config, token)` (re-scans one repo's `listOpenPulls`, pre-filtered on `auto_merge`/`labels`/`draft`) is still used by the `onInstallation`/`onReposAdded` install-event sweeps. (Those install handlers still load config per repo, so installing on a very large org can fan out ‚Äî a known, rare cost; deploy/cron/webhook all use the cheap search path.)

`maybeRetriggerZombiesForRepo` pre-filters open PRs to bot-authored ones (free, from `listOpenPulls`) before `reviveIfZombie` spends a KV read / API call, so re-sweeping an already-checked repo is nearly free; each PR is wrapped in try/catch. Cost is ~1 API call per *bot-authored* open PR not yet seen at its current SHA (App tokens get 5,000‚Äì12,500 req/hr).

Loop safety: our own close+reopen returns as a `reopened` event whose `sender.type` is `Bot` and is skipped ‚Äî the reliable synchronous guard (KV is eventually consistent, so the loop guard does **not** rely on it). `hasWorkflowRuns` fails safe to `true` on a query error, so transient failures never cause a spurious reopen. If the KV binding is ever absent, `reviveIfZombie` degrades to "always check, never record" (still correct, just not deduped) and `maybeBackfillRepo` no-ops.

`auto_open_pr`: `{ enabled, skip_branches[], skip_branch_patterns[], target_base, base_from_fork_point, base_branch_patterns[], close_when_empty, delete_branch_when_empty }` (default disabled). Root-cause fix for zombies: pr-minder itself opens PRs for forgotten branches, using its **App token** so they trigger CI natively (never zombies ‚Äî complementary to the `auto_trigger_workflows` band-aid). A branch qualifies when it's ahead of base **with a non-empty net diff** and has no open PR (`compareCommits` + `hasOpenPrForBranch` + `createPull` in `github.ts`; `shouldSkipBranch` always excludes the default branch + gh-pages). The non-empty-diff gate is load-bearing: after a squash-merge a branch is still "ahead" by commit *count* (squash shares no history) while its content already lives in base, so `compareCommits` now also returns `changed_files` (length of the comparison's `files` array, `null` when GitHub omits it ‚Üí fail open) and `maybeOpenPrForBranch` skips when `changed_files === 0`. Without it, pr-minder opened a **content-empty PR** whose `auto_describe_pr` then refused to describe ("no diff"), leaving it stuck with the branch name as its title/body forever (the #224/#225/#226 orphans). Two triggers: `onPushToBranch` (push to a non-default branch) and a sweep (`maybeOpenPrsForRepo`) on `installation`/`installation_repositories` for pre-existing branches. The base defaults to the branch's fork point (`base_from_fork_point` is **on by default** ‚Äî see below); `target_base` (default: repo default branch) is the fallback when no fork point is detected, or the base when `base_from_fork_point: false`. This replaces the "open PRs for branches missing one" reusable workflow in `wow-look-at-my/actions` (`pr-management.yml`), whose flaw was creating PRs with `github.token`.

**`close_when_empty`** (default **true**): the complement to the open gate. The open gate only stops pr-minder *opening* an empty PR ‚Äî but a PR opened with a real diff can go content-empty **after** the fact (its content lands in base another way, e.g. a sibling branch squash-merges the same change, or the branch's own commit is squashed into base), and the gate can't retract a PR that already exists. That's exactly how an orphan like `#243` slips through: it was opened ~90 min **before** the open gate (`#42`) deployed, as a squash-merge orphan (ahead by commit count, zero net diff); `#238` shows the post-open race (it *was* opened with a diff ‚Äî it got a real AI title ‚Äî then its content landed in `main` and it went empty). `closeEmptyAutoPrs()` cleans both up: for **every** open non-draft PR **regardless of author** (own bot, `github-actions[bot]`, any other bot, or a human ‚Äî closing is reversible, and a 0-diff PR has nothing to review or merge), it runs `compareCommits` and **closes (with a one-line comment) only when `changed_files === 0`** ‚Äî a `null`/unknown count is left alone, the same fail-open rule the open gate uses. (The author filter that once scoped this to pr-minder's own bot, then to bot authors, is gone: 0-diff = close.) It runs in `onPushToDefault` (the moment the base advances and a sibling PR can go empty) and on the `maybeBackfillRepo`/`onInstallation`/`onReposAdded` sweeps (mirroring where `maybeOpenPrsForRepo` runs). Set `close_when_empty: false` to opt out.

**`delete_branch_when_empty`** (default **false**): when `close_when_empty` closes a content-empty PR, also delete that PR's head branch (`deleteBranch()` in `github.ts`), so closed branches don't pile up. Two guards, both required: it only deletes a **same-repo head** (`pr.head.repo?.full_name === repo`), so a fork PR's head ‚Äî which lives in the fork and our token can't write ‚Äî is never touched; and `deleteBranch` treats GitHub `404` (already gone ‚Äî idempotent) and `422` (default/protected branch GitHub refuses to delete) as no-ops, so it never fails the close or fights branch protection (other failures throw, logged by `closeEmptyAutoPrs`'s per-PR try/catch). The branch is content-identical to base (zero net diff), so deleting it loses nothing. Off by default because deleting a branch is the one part of the cleanup that isn't trivially reversible; no effect unless `close_when_empty` is also on. Uses the App's existing `contents: write`.

`shouldSkipBranch` skips a branch when its name is the base, `gh-pages`, in `skip_branches` (exact), or matches a `skip_branch_patterns` regex (`matchesPattern`, which never throws on a bad pattern). The pattern list is what scalably excludes a *family* of branches ‚Äî e.g. an archive repo's hundreds of `^\d+\.\d+\.\d+$` version branches, which must never each become a PR head (one PR per version).

**Fork-point base detection** (`base_from_fork_point`, **default true** ‚Äî `mergeConfig`'s `result` initializer and `DISABLED` both seed it `true`; an explicit `base_from_fork_point: false` opts out): instead of always opening into `target_base`, `detectForkBase` (exported for tests) finds the branch the head was forked from and returns it with `ahead` (commits the head adds on top). **By default (`base_branch_patterns` empty) ANY other branch qualifies**, so a branch opens back into whatever branch it was actually forked from ‚Äî a branch forked off another working branch targets that working branch, not the default branch beneath it; the nearest fork point wins, and a tie at the same commit prefers the default branch (`names.includes(defaultBranch) ? defaultBranch : names[0]`). `base_branch_patterns`, when set, is a **restriction**: only the default branch or a pattern-matching branch may be a base (the `qualifies` predicate gates on `baseBranchPatterns.length === 0 || n === defaultBranch || matches`). That's the version-branch layout: a working branch forked from a long-lived version branch opens back into it, and the patterns keep an unrelated working-branch tip from being chosen instead. A branch forked off the default branch opens into the default branch; anything with no qualifying fork point falls back to `target_base`.

`detectForkBase` runs **two passes** so a parent that has *advanced past* the fork point is still found (the #47 follow-up ‚Äî a tip-only match silently fell back to the default branch once the parent moved): (1) **tip-walk** ‚Äî walk the head's commits newest-first (`listCommitShas`, capped at 2 pages) and match against `tipsBySha` (all branch tips from `listBranchHeads`); the nearest head-ancestor that is some qualifying branch's current tip is its fork point. This catches every parent still sitting at the fork point ‚Äî the common case, and crucially the **static long-lived branches** (archive version branches whose tip never moves), so they cost nothing extra. (2) **moved-parent scan** ‚Äî a parent that gained commits after the fork no longer has the fork commit as its tip, so pass 1 misses it, but the fork commit is still in the parent's recent *history*; for qualifying candidates **not** matched by their tip in pass 1, fetch their history (`listCommitShas`) and look for a fork commit nearer than the best tip-match (the newest of the candidate's commits that appears in the head's history = the merge base). The scan is **budget-bounded** (`FORK_SCAN_BUDGET`=10 per live call; a sweep shares one `SWEEP_FORK_SCAN_BUDGET`=30 so the install/backfill invocation stays under the subrequest cap), stops early once `best.ahead <= 1` (a 1-commit fork can't be beaten), and **skips static-tip parents entirely** (they were matched in pass 1) ‚Äî so the archive repo pays zero scan cost while an active integration branch that moved is still detected. Name the moving long-lived branches via `base_branch_patterns` to keep the scan small and exhaustive in a big repo; if the budget is exhausted detection degrades to the pass-1/`target_base` result (never crashes, never a worse base than before). The sweep builds the `tipsBySha` map **once** and passes it (plus the shared scan budget) to every `maybeOpenPrForBranch`; `onPushToBranch` builds it on demand. The fork-point `ahead` count is used only for logging; `maybeOpenPrForBranch` **always** runs `compareCommits` regardless, because the open gate needs the comparison's `changed_files` (emptiness can only be read from the real diff) ‚Äî one compare per about-to-open branch, a rare event.

`auto_describe_pr`: `{ enabled, model }` (default disabled) ‚Äî AI-generated PR titles/descriptions (`describe.ts` + the **pr-describe webhook** in `wow-look-at-my/webhooks`, served by webhook-runner). The model call routinely outlives the Worker's ~30s post-response grace, so the Worker only does the cheap half. On `opened` / `ready_for_review` / `synchronize` (`shouldDescribe`; drafts never reach it because `onPR` skips drafts wholesale), `maybeDescribePr` fetches the PR's **entire** base...head diff (`getPullDiff`, the `application/vnd.github.diff` media type) and hands it off **untruncated** ‚Äî there is no diff-size ceiling: an oversized diff is summarized in parts by the webhook (map-reduce, below), not capped here. `getPullDiff` also falls back to the **paginated files API** (reassembling a `diff --git` document from each file's `patch`) when GitHub returns **406** for a too-large unified diff, so even an enormous PR yields a diff. An **empty diff is no longer skipped silently** (the bug that left auto-opened orphans stuck with their branch name as the title): `markZeroDiff` stamps the PR `[zero diff] <branch>` (title + a short body) ‚Äî but **only for pr-minder's own bot PRs** (`appBotLogin`, so a human's PR is never relabeled; narrower than `closeEmptyAutoPrs`, which *closes* any author's empty PR ‚Äî a relabel mutates a title we may not own, whereas closing a verified-empty PR is safe and reversible, so the relabel stays own-bot-only) and **idempotently** (a title already starting with the marker isn't re-PATCHed). It's the on-event label; `auto_open_pr.close_when_empty` still closes such PRs on the next base update, and a later non-empty commit re-describes via the webhook. For a non-empty diff it dedups on the diff's SHA-256, and **POSTs the work to the webhook fire-and-forget**: `{repo, pr_number, old_title, old_body, diff, model, github_token}` to `DESCRIBE_HOOK_URL` (authed with the `DESCRIBE_HOOK_API_KEY` secret as `x-api-key`). webhook-runner answers 202 + a `run_id` immediately (it only spawns the container); the webhook then asks an OpenAI-compatible endpoint for metadata (temperature 0; system prompt + two bare user messages ‚Äî old title/description as JSON, then the diff). A diff at or below the webhook's single-call budget (`SINGLE_CALL_MAX_CHARS` ‚âà 110k) goes in **one** call (the original path); a larger one is split on file/hunk boundaries, **each part summarized in its own call, and the section summaries fused into the final metadata by a reduce call (map-reduce)** ‚Äî so arbitrarily large diffs still get a real description, just with more model calls. The webhook **retries transient model failures itself with backoff** (it has the hook's 15m timeout to work with), and PATCHes the PR: the **description is rewritten on every diff change**, the **title only when the model says the old one was a missing/autogenerated placeholder**.

**Superseded-run cancellation:** the returned `run_id` is stored in KV (`descrun:{repo}#{num}`, ~1 day TTL). When a newer diff hands off, the previous run may still be mid-model-call with stale input, so `maybeDescribePr` first POSTs `{DESCRIBE_HOOK_URL}/cancel/{run_id}` (best-effort ‚Äî 409 means it already finished, 404 evicted; network errors are logged and ignored) and only then submits the new run. That ordering means a stale PATCH can never land after a fresh one.

Endpoint config is split: the Worker holds `DESCRIBE_HOOK_URL` (plain var in wrangler.toml `[vars]`; if the feature is enabled and the var is absent, that's a loud misconfiguration ‚Äî see below), `PR_MINDER_PUBLIC_URL` (plain var in `[vars]` = the Worker's own public URL; `{this}/_describe-result` is the failure-callback endpoint, and like `DESCRIBE_HOOK_URL` it's **required** when `auto_describe_pr` is enabled ‚Äî a missing one throws, never a silent skip), and `DESCRIBE_HOOK_API_KEY` (secret = the hook's api_key, committed in plaintext in the webhooks repo's `pr-describe/hook.json`; doubles as the auth on the `/_describe-result` callback); the model endpoint/key (`AI_BASE_URL`/`AI_MODEL`/`AI_API_KEY`) live with the webhook in the webhooks repo's hook.json (+ runner-host env while `AI_API_KEY` is still a `${...}` reference). Per-repo config can override only `model` (rides along in the payload). Cost control: the hand-off is **deduped on a SHA-256 of the diff** in KV (`desc:{repo}#{num}`), recorded **optimistically** once the runner **accepts** the run (202) ‚Äî that dedup is what stops pr-minder's own `update-branch`-merge `synchronize` (and redeliveries) from re-describing an unchanged diff, costing one GitHub call and zero hand-offs. A failed *hand-off* (runner down, 4xx) throws with the marker untouched so the next event retries. A run that fails **after acceptance** (model error, PATCH failure) is the gap the optimistic record would otherwise leave open ‚Äî so the webhook **reports the terminal failure back** to `POST {PR_MINDER_PUBLIC_URL}/_describe-result` (authed with the shared `DESCRIBE_HOOK_API_KEY` as `x-api-key`), and `handleDescribeResult` clears the `desc:` marker (conditional on the reported diff hash, so a late callback from a superseded run can't wipe a newer describe ‚Äî `clearDescribedIfHash`), letting the next event re-describe instead of the PR being marked described forever. pr-minder **always** sends the callback fields (`fail_callback_url`, `fail_callback_key`, `diff_hash`) in the hand-off, and **requires** `PR_MINDER_PUBLIC_URL` when `auto_describe_pr` is enabled (a missing one is the same loud misconfiguration as a missing `DESCRIBE_HOOK_URL`, never a silent skip); the webhook in turn **requires** those fields on every payload (hard contract, so a run is always reportable). Latency control: `onPR` runs it through the `defer` callback (= `ctx.waitUntil`), so the webhook acks within GitHub's 10s window; failures never fail the webhook ‚Äî but they are **never silent** either.

**Failure visibility (`describeSafely`):** `onPR` calls `describeSafely`, which wraps `maybeDescribePr` and logs *any* error (missing `DESCRIBE_HOOK_URL` while enabled, hook 401 from a key mismatch, runner down/unreachable) at **error level** ‚Äî Workers Logs persists it (`[observability]` is enabled). Errors stay in the operator's panes by design: Worker-side failures in Workers Logs, and everything that reaches the runner (denied keys -> `hook.denied`, failed runs with output) on the runner's own dashboard ‚Äî **never as comments on the PR** (operator feedback: GitHub comments broadcast operational noise to every PR watcher). Rationale for the throw: the feature once shipped with "no URL -> log an info-level skip and no-op", which burned a full debugging round of "did the webhook even arrive?" ‚Äî enabled-but-misconfigured must be an error, not an opt-out.

**Backfill (the catch-up path, `runDescribeChecks`):** the live path above is the *only* describe trigger ‚Äî `opened`/`ready_for_review`/`synchronize` ‚Äî so a PR opened **before** `auto_describe_pr` shipped (or before the App was installed), or whose `opened`-event hand-off failed with **no `synchronize` since**, never gets a description: nothing revisits it, and it sits with its branch name as the title forever (the buildhost #63 case ‚Äî a `github-actions[bot]` PR from before the feature existed). Every *other* feature has a catch-up path; describe now has one too, built on the same enqueue-then-drain shape as `merge_conflict`. **Enqueue** (`enqueueDescribeChecks`/`enqueueDescribeChecksForRepo`, a cheap `describe:` KV put per candidate, no GitHub call beyond the cached `appBotLogin`): runs on a **default-branch push** (so an *already-installed, already-backfilled* repo's stuck PRs are picked up on its next merge ‚Äî the `backfill:` capability set, with `describe` already recorded swept, would otherwise gate them out), the **first-webhook backfill** (`maybeBackfillRepo`, gated by the `describe` capability), and the **install/repos-added sweeps** ‚Äî mirroring exactly where the conflict enqueue runs. A PR **qualifies** (`qualifiesForDescribeBackfill`) only when it's **bot-authored** (`github-actions[bot]` or pr-minder's own `appBotLogin`) **AND its title still equals its head branch name** ‚Äî the exact signature of an auto-opened PR that never got an AI title. That `title === head.ref` test is the load-bearing safety: a PR whose title was already set to something real (a prior describe, or a human's edit) no longer qualifies, so the backfill can **never overwrite a curated title/description** (the webhook rewrites the body on every run, so this gate is what protects human and dependabot descriptions ‚Äî both are also excluded by author / a non-branch title). **Drain** (`runDescribeChecks`, a cron pass alongside `runConflictChecks`, budget 10): reads only the reminders (a single KV list when none pending, zero GitHub calls), and per pending PR mints a token + loads config (both cached per run), **skips-and-clears** a PR that already has a `desc:` marker (described once ‚Äî the live path owns it from here; a cheap KV read, no diff fetch), is closed, has been retitled, or is in a now-feature-off repo, and otherwise calls `describeSafely` (itself idempotent on the diff hash, so a race with the live path can't double-describe) and clears the reminder. A transiently unreadable PR is left for the next tick (the `describe:` TTL ~1 day bounds an abandoned one). Because describing renames the title away from the branch, a described PR stops qualifying ‚Äî so the enqueue/drain is self-clearing and steady state is free.

There is no top-level `enabled`; "disable" means omitting the relevant label/mode, leaving `triggers` empty, or setting `auto_trigger_workflows: false` / `auto_open_pr.enabled: false` / `auto_describe_pr.enabled: false`.

## Key invariants

- `update-branch` returns 422 when already up to date ‚Äî this is not an error (handled in `github.ts`)
- **No empty merge commits.** GitHub's `update-branch` merges base into head whenever head is "behind" by commit *count*, even when head already contains base's *content* (a branch that tracks base closely, e.g. an uprev/sync branch, or a head that already superseded base's changes) ‚Äî leaving a 0-diff "Merge branch ..." commit on the PR. Every `update-branch` call therefore goes through `updateBranchUnlessEmpty(repo, pr, baseTipSha, ‚Ä¶)` (handlers.ts), which skips the merge when `mergeWouldBeEmpty(repo, pr, baseTipSha, ‚Ä¶)` (github.ts) says it would introduce nothing. The signal is **tree equality**, read from GitHub's own test-merge of the PR (`pull_request.merge_commit_sha`, kept at `refs/pull/N/merge` = base merged with head): a clean 3-way merge's *tree* is direction-independent, so that tree is exactly what `update-branch` (base into head) would produce, and the merge is empty iff it equals head's tree. `mergeWouldBeEmpty` fetches the merge commit's tree+parents and head's tree (two lightweight `GET /git/commits/{sha}` calls) and returns true iff `merge.tree === head.tree`. **It trusts the test-merge only when it's provably current** ‚Äî both the PR's current `head.sha` and the current base tip (`baseTipSha`) must be among the merge commit's parents (the ref is recomputed asynchronously, so a just-moved base can briefly leave it stale). Because the parents are verified, a "true" means the compared tree really is the merge of *these exact commits*: a wrong skip (dropping a real update) is impossible; any staleness/error/missing-`merge_commit_sha` yields false ‚Üí `update-branch` runs as before (worst case: a harmless extra merge, never a missed update). `baseTipSha` is supplied by the caller ‚Äî the push's `after` in `onPushToDefault`, the payload's `base.sha` in `onPR`. **Limitation:** right after a default-branch push GitHub may not have recomputed the test-merge yet (parents won't match the new base) ‚Äî that one update isn't skipped; the next read once the ref catches up is clean
- Auto-merge (`enableAutoMerge`/`disableAutoMerge`) is **GraphQL-only** ‚Äî there is no REST endpoint (`PUT/DELETE /repos/{repo}/pulls/{num}/automerge` returns 404). The mutations take the PR's `node_id` (from the webhook payload), not its number, and need contents:write + pull_requests:write. GraphQL logical failures arrive as HTTP 200 + an `errors[]` array (swallowed); only non-2xx transport failures throw.
- `pull_request_review` webhook sends `review.state` lowercase; the reviews REST API returns uppercase `APPROVED` ‚Äî both cases are handled
- PRs created via a workflow's default `GITHUB_TOKEN` never trigger their own workflows (GitHub's recursion guard); the author is `github-actions[bot]`. `auto_trigger_workflows` fixes this by close+reopen using the App's installation token (a different credential), which GitHub *does* let trigger workflows. Close+reopen is two `PATCH /repos/{repo}/pulls/{num}` calls (`state: closed` then `state: open`); `closed` is intentionally not a handled action. The retrigger runs on `opened`/`reopened`/`synchronize` (via `reviveIfZombie`, gated by `shouldConsiderRevive`) plus the first-webhook backfill and the install sweeps; it's deduped per-PR-per-commit by KV (`state.ts`). A follow-up commit is re-closed/reopened only after it ages past `REVIVE_MIN_AGE_S` without gaining runs, so pr-minder's own `update-branch` merge (a fresh head SHA with no runs yet) is deferred rather than re-triggered ‚Äî the commit-age guard, not the event sender, prevents the double cycle. The loop guard is the `Bot` sender skip on `reopened` (our own reopen comes back bot-sent), not the event gate and not the (eventually-consistent) KV state
- GitHub App must subscribe to `pull_request`, `pull_request_review`, and `push` events; `installation` and `installation_repositories` are auto-delivered. It should **not** be subscribed to anything else ‚Äî in particular **not** `workflow_job`/`workflow_run`, which a CI-heavy org emits in huge volume and pr-minder acts on none of. `handle()` gates on `HANDLED_EVENTS` and drops every other event type (incl. `ping`) up front ‚Äî before any token mint, KV access, or the auto-merge backstop ‚Äî so an over-broad subscription can't slam the Worker; but the real fix for a flood is to **unsubscribe the App from the unwanted events** in the App settings (GitHub then never delivers them, costing zero Worker invocations)
- `push` handling: default-branch pushes ‚Üí `onPushToDefault` (update qualifying PRs); non-default branch pushes ‚Üí `onPushToBranch` (`auto_open_pr`). (The zombie backfill for an already-installed repo rides the opportunistic per-event block in `handle()`, not `onPushToDefault` specifically, so *any* first webhook ‚Äî push or otherwise ‚Äî triggers it.) Both gated on `ref` being `refs/heads/*` and `!p.deleted`, so tag pushes and branch deletions are ignored. A PR pr-minder opens is authored by the App's own bot, so it triggers CI natively and the `auto_trigger_workflows` zombie check (which keys on `github-actions[bot]`) correctly ignores it ‚Äî no interaction between the two features
- JWT validity window is `iat - 60s` to `exp + 540s` (GitHub allows up to 10 min; we use 9)
- `auto_describe_pr` cannot loop: the pr-describe webhook's PATCH fires `pull_request.edited`, which is not a handled action, and the edit doesn't change the head SHA (no `synchronize`). The KV diff-hash is an *optimization*, not the loop guard
- The `defer` callback (ctx.waitUntil) means describe hand-off logs land after the webhook response ‚Äî visible in observability (`console.log`), not in the delivery's response body. The model call's own logs live on the webhook-runner side (the run's output in its dashboard)
