# pr-minder

Cloudflare Worker (TypeScript) that acts as a GitHub App webhook handler for keeping PRs up to date.

## Source layout

```
src/
  worker.ts     Entry point: Cloudflare Worker. fetch(): GET/HEAD -> serveDocs(), POST /_describe-result -> handleDescribeResult (the pr-describe failure callback), other POST -> webhook; on each isolate's first request it kicks startupReconcile() once via ctx.waitUntil (reconcile-on-startup, not polling). scheduled(): the cron entry point -> runRechecks() (drains the deferred-revive reminders) + reconcileAllInstalls() (the auto-merge backstop)
  describe-result.ts  handleDescribeResult(): the POST /_describe-result endpoint — the pr-describe webhook reports a terminally failed run, this clears the `desc:` marker (via clearDescribedIfHash) so the next event re-describes. Own module (not worker.ts) so tests import it without worker.ts's gzipped-docs blobs — same reason as webhook.ts. Auth: shared DESCRIBE_HOOK_API_KEY via x-api-key (timingSafeEqual)
  webhook.ts    verifyWebhook(): HMAC-SHA256 check of x-hub-signature-256 (its own module so tests don't pull in worker.ts's baked-in docs)
  handlers.ts   Event dispatch: handle() (-> dispatch() then the cooldown-gated auto-merge backstop), onPR(), onPushToDefault(), onPushToBranch(), onInstallation(), onReposAdded(), prQualifies(), isActionsBotPr(), shouldConsiderRevive(), reviveIfZombie(), runRechecks(), maybeBackfillRepo(), startupReconcile(), reconcileAllInstalls(), reconcileInstall() (search-based auto-merge backstop), reconcileAutoMerge() (per-repo, install events only), shouldSkipBranch(), detectForkBase() (auto_open_pr fork-point base selection), closeEmptyAutoPrs() (close content-empty PRs pr-minder itself opened — runs on default-branch push + the install/backfill sweeps). handle/dispatch/onPR take an optional `defer` callback (worker.ts passes ctx.waitUntil) that runs the auto_describe_pr hand-off after the webhook response

  docs/         Doc sources: llms.txt (the content) and index.html (a self-contained page that fetches /llms.txt and renders it). *.gz are generated, git-ignored
```

## Docs serving

`worker.ts` serves public docs on GET/HEAD: `/` and `/llms.txt`. `llms.txt` is the single source of truth; `index.html` is a self-contained page that fetches `/llms.txt` and renders the Markdown. Both are **gzipped at build time** (`scripts/build-docs.mjs`, wired via `wrangler.toml [build].command`, output git-ignored) and imported as `*.gz` blobs (the `[[rules]] type = "Data"` makes them `ArrayBuffer`). Serving negotiates on `Accept-Encoding` (and sets `Vary: Accept-Encoding`): clients that accept gzip get the blob as-is — `new Response(gz, { encodeBody: "manual", headers: { "content-encoding": "gzip" } })`, where `encodeBody: "manual"` is required else the runtime recompresses the already-gzipped bytes; clients that don't get the blob decompressed server-side via `DecompressionStream` and served identity. Keep doc content in `docs/*` (the "no embedded docs in source files" rule); never inline it in a `.ts`. Install URL: `https://github.com/apps/pr-minder/installations/new`. Webhooks are POST, so they never collide with the GET docs routes. `verifyWebhook` lives in `webhook.ts` so the test suite doesn't pull `worker.ts`'s binary imports through vite.

## Deployment — NEVER deploy manually

It is **never** valid to deploy by hand. Do NOT run `npm run deploy`,
`wrangler deploy`, or any other deploy command — not even to "test" or "ship a
fix faster". Cloudflare's Git integration watches this repo and deploys
automatically: a preview Worker for every PR branch, and **production on merge to
`master`** (the repo's default branch). Manual deploys bypass that pipeline and are
never the right move — ship by merging the PR.

Set via `wrangler secret put <NAME>`:

- `GITHUB_APP_PRIVATE_KEY` — PKCS8 PEM for the GitHub App private key
- `DESCRIBE_HOOK_API_KEY` — api_key of the pr-describe webhook (must equal the `api_key` committed in plaintext in the webhooks repo's `pr-describe/hook.json` — private repo, operator's call). The hook URL itself is the plain `DESCRIBE_HOOK_URL` var, **committed in wrangler.toml `[vars]`** — never a dashboard-only var, because each deploy replaces the script's plain vars with the `[vars]` block (dashboard-added vars silently vanish; secrets survive). (The model endpoint's key lives on the webhook side, not here)

## Config file loading order

1. `{repo}/.github/config/pr-minder/pr-minder.jsonc` — per-repo (highest priority)
2. `{org}/.github` repo, `.github/config/pr-minder/pr-minder.jsonc` (same path) → `repos.{repo}` field for per-repo overrides, top-level for org defaults
3. No config found → everything disabled (opt-in design; nothing fires by default)

`loadConfig` runs on essentially every webhook event and each resolution is up to two Contents API calls (per-repo file, then the org `.github` file — the common case is a 404 + a hit/404). To avoid hammering the API for config that rarely changes, the resolved result is **memoized per `owner/repo` for 60s** (`CONFIG_CACHE_TTL_MS`) in a module-scope `Map` — same per-isolate-lifetime pattern as `handlers.ts`'s `labelCheckedAt`. Consequences: a config edit takes up to 60s to take effect, and the cache key is `owner/repo` only (the resolved config is identical regardless of which install token reads it). Only **definitive** resolutions are cached: a transient fetch failure (5xx/throttling/network — `fetchRepoFile` throws on any non-2xx that isn't a 404) degrades to "disabled" for that one event but is **not** cached, so the next event retries instead of being stuck with an empty config for a full TTL. A 404 (file absent) and a malformed-JSON file are definitive and *are* cached.

Separately, the **org `.github` file is memoized per `owner`** (`orgConfigCache`, also 60s) inside `loadOrgConfig`, because that one file is shared by every repo in the org. The `owner/repo` cache above only de-dupes repeated reads of the *same* repo; it does nothing for a pass over many *different* repos in one org. Without the owner-level cache, a cross-owner sweep re-fetched the shared org file once per repo — the dominant cost that, at ~380 repos, blew Cloudflare's 50-external-subrequest cap in a single invocation. Same cacheability rule: a 404/malformed org file is cached (as "no org config"), a transient fetch failure is not. `loadOwnerConfig(owner)` returns just the org-level config (no per-repo file or override) and is what the auto-merge backstop uses to learn the `auto_merge` label names for its search.

## Config shape

`auto_update_pr.triggers`: approval-gate conditions; any one passing fires `update-branch` (skipped when the merge would be a no-op — see "No empty merge commits" under Key invariants). Keys within a condition are ANDed; conditions are ORed. Supported keys: `label`, `approved_by`, `min_approvals`.

`auto_label_pr`: map of label name → `{ auto_add, create_label_if_missing_in_repo, color, mode, auto_merge_method }`.

- `auto_add: "on_pr_creation"` — apply label when a PR is opened.
- `create_label_if_missing_in_repo: true` — create the label in the repo (with `color`) if absent.
- `mode: "auto_update"` — when this label is present on a PR, keep the branch in sync with its base (complementary to `auto_update_pr.triggers`).
- `mode: "auto_merge"` — bidirectionally syncs this label with GitHub's native auto-merge. Adding the label enables auto-merge; removing it disables auto-merge. `auto_merge_enabled`/`auto_merge_disabled` webhook events add/remove the label. **Whenever GitHub refuses to *arm* native auto-merge** (`enablePullRequestAutoMerge` returns HTTP 200 + an `errors[]` array), `enableAutoMerge` falls back to merging the PR directly via `mergePullRequest` (REST `PUT .../merge`) with `auto_merge_method`. The two cases that hit this: the PR is **already mergeable** (`"clean status"` — nothing pending for auto-merge to wait on) and the repo has **not enabled "Allow auto-merge"** in its settings (`"Auto merge is not allowed for this repository"` — pr-minder has no `administration` permission, so it can't toggle that setting itself). The fallback is unconditional on logical-error responses precisely so the label means "merge this" even in repos without native auto-merge enabled; `mergePullRequest` is the final authority, so branch protection still gates the merge and a not-yet-mergeable PR comes back as a swallowed 4xx (never merged). When native auto-merge *can* be armed, the success path returns first, so pr-minder never pre-empts GitHub's wait-for-checks behavior. (Limitation: in a repo without native auto-merge, a labeled PR with pending required checks won't be merged later automatically — pr-minder isn't subscribed to check/status completion events — so enable "Allow auto-merge" on the repo if you want the wait-for-checks behavior.)
- `auto_merge_method` — `"merge"`, `"squash"` (default), or `"rebase"`. Only used when `mode` is `"auto_merge"`.

`auto_trigger_workflows`: boolean (default false). Revives "zombie" PRs — author `github-actions[bot]` (created with the default `GITHUB_TOKEN`, whose events GitHub won't let trigger workflows) and no workflow runs for their head commit — by closing+reopening with the App token (`retriggerWorkflows` in `github.ts`), which fires a fresh event that DOES run their CI. Gated on `auto_trigger_workflows` + bot-author throughout.

One core check, `reviveIfZombie(env, repo, pr, …)`, is shared by every trigger: bot-author-gated, it spends a `hasWorkflowRuns` call (`GET /actions/runs?head_sha=…`) and close+reopens if there are none. It is **check-once via Workers KV** (`state.ts`): keyed `pr:{repo}#{num}` → the head SHA last evaluated; if the current SHA matches it's skipped, else it's evaluated and the SHA recorded. A new commit changes the SHA, so a *touched* PR is re-checked while an untouched one never is. **The first commit we ever handle for a PR (no prior SHA recorded) is revived immediately** — a bot PR is born with zero CI, so that reading is trustworthy. **A *follow-up* commit with zero runs is only revived once it has aged past `REVIVE_MIN_AGE_S` (~60s, in `handlers.ts`)**, fetched via `commitAgeSeconds` (`GET /repos/{repo}/commits/{sha}`); a still-fresh one is left *unrecorded* and a `recheck:{repo}#{num}` reminder is dropped in KV (`setRecheck`), because webhooks won't re-fire on their own once the commit ages. This is what stops pr-minder's **own `update-branch` merge** — a new head SHA whose runs haven't registered yet — from being close+reopened a second time, without needing to know who pushed it. (On a commit-age lookup error `commitAgeSeconds` returns null and the follow-up is deferred, failing safe against a spurious re-close.) The reminder is drained by `runRechecks` (next bullet); on a verdict (revived, or runs found) the reminder is cleared.

Triggers (event-driven, plus one narrow cron for deferred re-checks):
- Live `pull_request` **opened / reopened / synchronize** → `reviveIfZombie` on that PR, gated by `shouldConsiderRevive(action, sender)` (all three actions qualify; the only one dropped is a `reopened` whose `sender.type === 'Bot'` — our own close+reopen coming back — so the cycle can't loop). `synchronize` is included so new commits get re-checked; the **commit-age guard inside `reviveIfZombie`** (not the event sender) is what keeps pr-minder's own `update-branch` merge from being closed+reopened a *second* time — that merge is a follow-up commit still too fresh to have registered CI, so it's deferred. If it reopened, `onPR` returns and the fresh event drives the rest.
- **Cron, every 5 min** (the Worker's `scheduled` handler) → two cheap passes: (1) `runRechecks` drains the `recheck:` reminders left by deferred follow-up commits, re-evaluating each PR once it has aged. This is **not a poll over all PRs**: it reads only the reminders (`listRechecks`, a KV list), so with none pending it makes a single KV call and **zero GitHub API calls**. Per pending PR it mints a token for that repo (`repoInstallationId` → `installToken`, cached per run), refetches the PR (`getPull`), and re-runs `reviveIfZombie`. Closed/merged PRs clear their reminder; a missing token or transient fetch failure leaves it for the next sweep (KV TTL ~1 day bounds an abandoned one). (2) `reconcileAllInstalls` runs the auto-merge backstop across every installation (see below).
- **First webhook from a repo** → `maybeBackfillRepo` (run from the opportunistic per-event block in `handle()`): the event-driven "check at least once" for repos installed before the feature existed (GitHub never re-sends their install event). On a repo's first event of any kind it sweeps the repo's open PRs once and sets a `backfill:{repo}` KV flag; every later event is then a single KV read.
- `installation`/`installation_repositories` → the same `maybeRetriggerZombiesForRepo` sweep for the install's repos, then sets the backfill flag.

Auto-merge backstop (`reconcileInstall` / `reconcileAllInstalls`): the safety net for an `auto_merge`-labeled PR the live webhook path dropped (e.g. a webhook whose invocation hit the subrequest cap — the original failure mode that left PRs silently unmerged, no error, HTTP 200). `reconcileInstall(owner, token, log, budget)` loads the **owner-level** config (`loadOwnerConfig`) for the `auto_merge` label names, then for each label **searches the whole installation** (`searchPrsByLabel` → `GET /search/issues?q=is:pr is:open draft:false label:"…"`; an install token scopes results to the install's repos, so no `org:` qualifier) and on each hit not already armed calls `enableAutoMerge` (which direct-merges a PR that's already mergeable, so a "ready" PR gets merged instead of sitting forever). **Cost scales with *labeled* PRs, not repo count** — one search covers an owner's entire fleet (a 500-PR repo with 3 labeled PRs returns 3), which is what makes this safe to run fleet-wide in a single invocation. Per-PR work needs a `getPull` (to read `node_id`/`auto_merge`) and is wrapped in try/catch; `budget.calls` bounds the GitHub calls so an invocation stays well under the 50-external-subrequest cap, with anything not reached picked up next time.

It runs in three places: (1) **deploy** — `startupReconcile(env)` calls `reconcileAllInstalls` (iterate installs → token → `reconcileInstall`), still **gated once per deploy** by the version KV flag (`startup:{CF_VERSION_METADATA.id}`), claimed *before* sweeping so a concurrent isolate bails; fired from `worker.ts` on the first request via `ctx.waitUntil` + a per-isolate module flag. (2) **cron, every 5 min** — `reconcileAllInstalls` again (a fresh invocation, larger budget). (3) **every webhook** — after `dispatch`, `handle()` runs `reconcileInstall` for the webhook's owner, **cooldown-gated per owner** via a `swept:{owner}` KV marker (`recentlySwept`/`markSwept`, ~60s TTL) so a burst of webhooks can't exceed GitHub's ~30/min search rate limit; the marker is claimed before running so concurrent isolates don't pile on, and failures are swallowed (the backstop must never fail the webhook). This is self-feeding: a merge here emits its own push webhook, which drives the next pass. Steady state is still driven by the live `labeled`/`unlabeled`/`auto_merge_enabled`/`auto_merge_disabled` webhooks (immediate); the backstop only catches what they dropped.

The older per-repo `reconcileAutoMerge(repo, config, token)` (re-scans one repo's `listOpenPulls`, pre-filtered on `auto_merge`/`labels`/`draft`) is still used by the `onInstallation`/`onReposAdded` install-event sweeps. (Those install handlers still load config per repo, so installing on a very large org can fan out — a known, rare cost; deploy/cron/webhook all use the cheap search path.)

`maybeRetriggerZombiesForRepo` pre-filters open PRs to bot-authored ones (free, from `listOpenPulls`) before `reviveIfZombie` spends a KV read / API call, so re-sweeping an already-checked repo is nearly free; each PR is wrapped in try/catch. Cost is ~1 API call per *bot-authored* open PR not yet seen at its current SHA (App tokens get 5,000–12,500 req/hr).

Loop safety: our own close+reopen returns as a `reopened` event whose `sender.type` is `Bot` and is skipped — the reliable synchronous guard (KV is eventually consistent, so the loop guard does **not** rely on it). `hasWorkflowRuns` fails safe to `true` on a query error, so transient failures never cause a spurious reopen. If the KV binding is ever absent, `reviveIfZombie` degrades to "always check, never record" (still correct, just not deduped) and `maybeBackfillRepo` no-ops.

`auto_open_pr`: `{ enabled, skip_branches[], skip_branch_patterns[], target_base, base_from_fork_point, base_branch_patterns[], close_when_empty }` (default disabled). Root-cause fix for zombies: pr-minder itself opens PRs for forgotten branches, using its **App token** so they trigger CI natively (never zombies — complementary to the `auto_trigger_workflows` band-aid). A branch qualifies when it's ahead of base **with a non-empty net diff** and has no open PR (`compareCommits` + `hasOpenPrForBranch` + `createPull` in `github.ts`; `shouldSkipBranch` always excludes the default branch + gh-pages). The non-empty-diff gate is load-bearing: after a squash-merge a branch is still "ahead" by commit *count* (squash shares no history) while its content already lives in base, so `compareCommits` now also returns `changed_files` (length of the comparison's `files` array, `null` when GitHub omits it → fail open) and `maybeOpenPrForBranch` skips when `changed_files === 0`. Without it, pr-minder opened a **content-empty PR** whose `auto_describe_pr` then refused to describe ("no diff"), leaving it stuck with the branch name as its title/body forever (the #224/#225/#226 orphans). Two triggers: `onPushToBranch` (push to a non-default branch) and a sweep (`maybeOpenPrsForRepo`) on `installation`/`installation_repositories` for pre-existing branches. `target_base` defaults to the repo default branch. This replaces the "open PRs for branches missing one" reusable workflow in `wow-look-at-my/actions` (`pr-management.yml`), whose flaw was creating PRs with `github.token`.

**`close_when_empty`** (default **true**): the complement to the open gate. The open gate only stops pr-minder *opening* an empty PR — but a PR opened with a real diff can go content-empty **after** the fact (its content lands in base another way, e.g. a sibling branch squash-merges the same change, or the branch's own commit is squashed into base), and the gate can't retract a PR that already exists. That's exactly how an orphan like `#243` slips through: it was opened ~90 min **before** the open gate (`#42`) deployed, as a squash-merge orphan (ahead by commit count, zero net diff); `#238` shows the post-open race (it *was* opened with a diff — it got a real AI title — then its content landed in `main` and it went empty). `closeEmptyAutoPrs()` cleans both up: for each open PR **authored by pr-minder's own App bot** (`appBotLogin()` → `{slug}[bot]` — a human's PR is never touched), it runs `compareCommits` and **closes (with a one-line comment) only when `changed_files === 0`** — a `null`/unknown count is left alone, the same fail-open rule the open gate uses. It runs in `onPushToDefault` (the moment the base advances and a sibling PR can go empty) and on the `maybeBackfillRepo`/`onInstallation`/`onReposAdded` sweeps (mirroring where `maybeOpenPrsForRepo` runs). Set `close_when_empty: false` to opt out.

`shouldSkipBranch` skips a branch when its name is the base, `gh-pages`, in `skip_branches` (exact), or matches a `skip_branch_patterns` regex (`matchesPattern`, which never throws on a bad pattern). The pattern list is what scalably excludes a *family* of branches — e.g. an archive repo's hundreds of `^\d+\.\d+\.\d+$` version branches, which must never each become a PR head (one PR per version).

**Fork-point base detection** (`base_from_fork_point`): instead of always opening into `target_base`, `detectForkBase` (exported for tests) lists the branch's commits newest-first (`listCommitShas`, capped at 2 pages) and returns the first ancestor SHA that is the HEAD of another *qualifying* branch — the default branch, or a non-default branch matching `base_branch_patterns` — along with `ahead` (that ancestor's index = commits added on top). Built for the version-branch layout: a working branch forked from a long-lived version branch (whose tip never moves, so it *is* the fork point) opens back into that version branch; a branch forked off the default branch opens into the default branch; anything with no qualifying fork point falls back to `target_base`. The SHA→branch-name lookup comes from `listBranchHeads` (replaces the old name-only `listBranches`); the sweep builds that `tipsBySha` map **once** and passes it to every `maybeOpenPrForBranch` (a single `listCommitShas` per non-skipped branch), while `onPushToBranch` builds it on demand only when the feature is on. The fork-point `ahead` count is used only for logging; `maybeOpenPrForBranch` **always** runs `compareCommits` regardless, because the open gate now needs the comparison's `changed_files` (emptiness can only be read from the real diff) — one compare per about-to-open branch, a rare event.

`auto_describe_pr`: `{ enabled, model }` (default disabled) — AI-generated PR titles/descriptions (`describe.ts` + the **pr-describe webhook** in `wow-look-at-my/webhooks`, served by webhook-runner). The model call routinely outlives the Worker's ~30s post-response grace, so the Worker only does the cheap half. On `opened` / `ready_for_review` / `synchronize` (`shouldDescribe`; drafts never reach it because `onPR` skips drafts wholesale), `maybeDescribePr` fetches the PR's **entire** base...head diff (`getPullDiff`, the `application/vnd.github.diff` media type) and hands it off **untruncated** — there is no diff-size ceiling: an oversized diff is summarized in parts by the webhook (map-reduce, below), not capped here. `getPullDiff` also falls back to the **paginated files API** (reassembling a `diff --git` document from each file's `patch`) when GitHub returns **406** for a too-large unified diff, so even an enormous PR yields a diff. An **empty diff is no longer skipped silently** (the bug that left auto-opened orphans stuck with their branch name as the title): `markZeroDiff` stamps the PR `[zero diff] <branch>` (title + a short body) — but **only for pr-minder's own bot PRs** (`appBotLogin`, mirroring `closeEmptyAutoPrs`'s safety rule, so a human's PR is never relabeled) and **idempotently** (a title already starting with the marker isn't re-PATCHed). It's the on-event label; `auto_open_pr.close_when_empty` still closes such PRs on the next base update, and a later non-empty commit re-describes via the webhook. For a non-empty diff it dedups on the diff's SHA-256, and **POSTs the work to the webhook fire-and-forget**: `{repo, pr_number, old_title, old_body, diff, model, github_token}` to `DESCRIBE_HOOK_URL` (authed with the `DESCRIBE_HOOK_API_KEY` secret as `x-api-key`). webhook-runner answers 202 + a `run_id` immediately (it only spawns the container); the webhook then asks an OpenAI-compatible endpoint for metadata (temperature 0; system prompt + two bare user messages — old title/description as JSON, then the diff). A diff at or below the webhook's single-call budget (`SINGLE_CALL_MAX_CHARS` ≈ 110k) goes in **one** call (the original path); a larger one is split on file/hunk boundaries, **each part summarized in its own call, and the section summaries fused into the final metadata by a reduce call (map-reduce)** — so arbitrarily large diffs still get a real description, just with more model calls. The webhook **retries transient model failures itself with backoff** (it has the hook's 15m timeout to work with), and PATCHes the PR: the **description is rewritten on every diff change**, the **title only when the model says the old one was a missing/autogenerated placeholder**.

**Superseded-run cancellation:** the returned `run_id` is stored in KV (`descrun:{repo}#{num}`, ~1 day TTL). When a newer diff hands off, the previous run may still be mid-model-call with stale input, so `maybeDescribePr` first POSTs `{DESCRIBE_HOOK_URL}/cancel/{run_id}` (best-effort — 409 means it already finished, 404 evicted; network errors are logged and ignored) and only then submits the new run. That ordering means a stale PATCH can never land after a fresh one.

Endpoint config is split: the Worker holds `DESCRIBE_HOOK_URL` (plain var in wrangler.toml `[vars]`; if the feature is enabled and the var is absent, that's a loud misconfiguration — see below), `PR_MINDER_PUBLIC_URL` (plain var in `[vars]` = the Worker's own public URL; `{this}/_describe-result` is the failure-callback endpoint, and like `DESCRIBE_HOOK_URL` it's **required** when `auto_describe_pr` is enabled — a missing one throws, never a silent skip), and `DESCRIBE_HOOK_API_KEY` (secret = the hook's api_key, committed in plaintext in the webhooks repo's `pr-describe/hook.json`; doubles as the auth on the `/_describe-result` callback); the model endpoint/key (`AI_BASE_URL`/`AI_MODEL`/`AI_API_KEY`) live with the webhook in the webhooks repo's hook.json (+ runner-host env while `AI_API_KEY` is still a `${...}` reference). Per-repo config can override only `model` (rides along in the payload). Cost control: the hand-off is **deduped on a SHA-256 of the diff** in KV (`desc:{repo}#{num}`), recorded **optimistically** once the runner **accepts** the run (202) — that dedup is what stops pr-minder's own `update-branch`-merge `synchronize` (and redeliveries) from re-describing an unchanged diff, costing one GitHub call and zero hand-offs. A failed *hand-off* (runner down, 4xx) throws with the marker untouched so the next event retries. A run that fails **after acceptance** (model error, PATCH failure) is the gap the optimistic record would otherwise leave open — so the webhook **reports the terminal failure back** to `POST {PR_MINDER_PUBLIC_URL}/_describe-result` (authed with the shared `DESCRIBE_HOOK_API_KEY` as `x-api-key`), and `handleDescribeResult` clears the `desc:` marker (conditional on the reported diff hash, so a late callback from a superseded run can't wipe a newer describe — `clearDescribedIfHash`), letting the next event re-describe instead of the PR being marked described forever. pr-minder **always** sends the callback fields (`fail_callback_url`, `fail_callback_key`, `diff_hash`) in the hand-off, and **requires** `PR_MINDER_PUBLIC_URL` when `auto_describe_pr` is enabled (a missing one is the same loud misconfiguration as a missing `DESCRIBE_HOOK_URL`, never a silent skip); the webhook in turn **requires** those fields on every payload (hard contract, so a run is always reportable). Latency control: `onPR` runs it through the `defer` callback (= `ctx.waitUntil`), so the webhook acks within GitHub's 10s window; failures never fail the webhook — but they are **never silent** either.

**Failure visibility (`describeSafely`):** `onPR` calls `describeSafely`, which wraps `maybeDescribePr` and logs *any* error (missing `DESCRIBE_HOOK_URL` while enabled, hook 401 from a key mismatch, runner down/unreachable) at **error level** — Workers Logs persists it (`[observability]` is enabled). Errors stay in the operator's panes by design: Worker-side failures in Workers Logs, and everything that reaches the runner (denied keys -> `hook.denied`, failed runs with output) on the runner's own dashboard — **never as comments on the PR** (operator feedback: GitHub comments broadcast operational noise to every PR watcher). Rationale for the throw: the feature once shipped with "no URL -> log an info-level skip and no-op", which burned a full debugging round of "did the webhook even arrive?" — enabled-but-misconfigured must be an error, not an opt-out.
There is no top-level `enabled`; "disable" means omitting the relevant label/mode, leaving `triggers` empty, or setting `auto_trigger_workflows: false` / `auto_open_pr.enabled: false` / `auto_describe_pr.enabled: false`.

## Key invariants

- `update-branch` returns 422 when already up to date — this is not an error (handled in `github.ts`)
- **No empty merge commits.** GitHub's `update-branch` merges base into head whenever head is "behind" by commit *count*, even when head already contains base's *content* (a branch that tracks base closely, e.g. an uprev/sync branch, or a head that already superseded base's changes) — leaving a 0-diff "Merge branch ..." commit on the PR. Every `update-branch` call therefore goes through `updateBranchUnlessEmpty(repo, pr, baseTipSha, …)` (handlers.ts), which skips the merge when `mergeWouldBeEmpty(repo, pr, baseTipSha, …)` (github.ts) says it would introduce nothing. The signal is **tree equality**, read from GitHub's own test-merge of the PR (`pull_request.merge_commit_sha`, kept at `refs/pull/N/merge` = base merged with head): a clean 3-way merge's *tree* is direction-independent, so that tree is exactly what `update-branch` (base into head) would produce, and the merge is empty iff it equals head's tree. `mergeWouldBeEmpty` fetches the merge commit's tree+parents and head's tree (two lightweight `GET /git/commits/{sha}` calls) and returns true iff `merge.tree === head.tree`. **It trusts the test-merge only when it's provably current** — both the PR's current `head.sha` and the current base tip (`baseTipSha`) must be among the merge commit's parents (the ref is recomputed asynchronously, so a just-moved base can briefly leave it stale). Because the parents are verified, a "true" means the compared tree really is the merge of *these exact commits*: a wrong skip (dropping a real update) is impossible; any staleness/error/missing-`merge_commit_sha` yields false → `update-branch` runs as before (worst case: a harmless extra merge, never a missed update). `baseTipSha` is supplied by the caller — the push's `after` in `onPushToDefault`, the payload's `base.sha` in `onPR`. **Limitation:** right after a default-branch push GitHub may not have recomputed the test-merge yet (parents won't match the new base) — that one update isn't skipped; the next read once the ref catches up is clean
- Auto-merge (`enableAutoMerge`/`disableAutoMerge`) is **GraphQL-only** — there is no REST endpoint (`PUT/DELETE /repos/{repo}/pulls/{num}/automerge` returns 404). The mutations take the PR's `node_id` (from the webhook payload), not its number, and need contents:write + pull_requests:write. GraphQL logical failures arrive as HTTP 200 + an `errors[]` array (swallowed); only non-2xx transport failures throw.
- `pull_request_review` webhook sends `review.state` lowercase; the reviews REST API returns uppercase `APPROVED` — both cases are handled
- PRs created via a workflow's default `GITHUB_TOKEN` never trigger their own workflows (GitHub's recursion guard); the author is `github-actions[bot]`. `auto_trigger_workflows` fixes this by close+reopen using the App's installation token (a different credential), which GitHub *does* let trigger workflows. Close+reopen is two `PATCH /repos/{repo}/pulls/{num}` calls (`state: closed` then `state: open`); `closed` is intentionally not a handled action. The retrigger runs on `opened`/`reopened`/`synchronize` (via `reviveIfZombie`, gated by `shouldConsiderRevive`) plus the first-webhook backfill and the install sweeps; it's deduped per-PR-per-commit by KV (`state.ts`). A follow-up commit is re-closed/reopened only after it ages past `REVIVE_MIN_AGE_S` without gaining runs, so pr-minder's own `update-branch` merge (a fresh head SHA with no runs yet) is deferred rather than re-triggered — the commit-age guard, not the event sender, prevents the double cycle. The loop guard is the `Bot` sender skip on `reopened` (our own reopen comes back bot-sent), not the event gate and not the (eventually-consistent) KV state
- GitHub App must subscribe to `pull_request`, `pull_request_review`, and `push` events; `installation` and `installation_repositories` are auto-delivered
- `push` handling: default-branch pushes → `onPushToDefault` (update qualifying PRs); non-default branch pushes → `onPushToBranch` (`auto_open_pr`). (The zombie backfill for an already-installed repo rides the opportunistic per-event block in `handle()`, not `onPushToDefault` specifically, so *any* first webhook — push or otherwise — triggers it.) Both gated on `ref` being `refs/heads/*` and `!p.deleted`, so tag pushes and branch deletions are ignored. A PR pr-minder opens is authored by the App's own bot, so it triggers CI natively and the `auto_trigger_workflows` zombie check (which keys on `github-actions[bot]`) correctly ignores it — no interaction between the two features
- JWT validity window is `iat - 60s` to `exp + 540s` (GitHub allows up to 10 min; we use 9)
- `auto_describe_pr` cannot loop: the pr-describe webhook's PATCH fires `pull_request.edited`, which is not a handled action, and the edit doesn't change the head SHA (no `synchronize`). The KV diff-hash is an *optimization*, not the loop guard
- The `defer` callback (ctx.waitUntil) means describe hand-off logs land after the webhook response — visible in observability (`console.log`), not in the delivery's response body. The model call's own logs live on the webhook-runner side (the run's output in its dashboard)
