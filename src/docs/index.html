pr-minder

A GitHub App that keeps your pull requests healthy — no per-repo workflows, no fuss.

Install the GitHub App →

pr-minder runs as a Cloudflare Worker and reacts to GitHub webhooks in about a second. It keeps PR branches up to date with their base, syncs labels with GitHub's native auto-merge, revives “zombie” PRs whose CI never ran, and opens PRs for forgotten branches. It is opt-in and purely event-driven — no database, no cron.

Install

Install the app and choose All repositories or a selected set:

github.com/apps/pr-minder/installations/new

Installing is safe: pr-minder does nothing until you add a config file. Every behavior is opt-in.

How it works

pr-minder receives the pull_request, pull_request_review, push, installation, and installation_repositories webhooks. For each one it verifies the signature, mints a short-lived installation token, loads your config, and acts through the GitHub REST and GraphQL APIs. It stores no state.

Because it acts with the App's own credentials (not a workflow's default GITHUB_TOKEN), the PRs it opens and reopens trigger their workflows normally — which is what makes the zombie-PR features below possible.

Configuration

Config files are JSONC — // and /* */ comments are allowed. pr-minder resolves config in this order (first match wins):

  1. <repo>/.github/pr-minder.jsonc — per-repo config (highest priority).
  2. <org>/.github repo, at .github/config/pr-minder/pr-minder.jsonc — org-wide defaults, with per-repo overrides under repos.<repo-name>.
  3. No config found — everything is disabled.

Reference the schema for editor validation and autocomplete:

{
  "$schema": "https://raw.githubusercontent.com/wow-look-at-my/pr-minder/master/schema/pr-minder.schema.json"
}

There is no global on/off switch — a feature is active only when its section is present and configured.

auto_update_pr — keep PR branches up to date

auto_update_pr.triggers is a list of conditions. When any condition holds, pr-minder updates the PR branch with its base. Keys within one condition are ANDed; conditions in the list are ORed.

KeyMeaning
labelPR has this label.
approved_byPR is approved by at least one of these usernames.
min_approvalsPR has at least this many approvals (from anyone).
{
  "auto_update_pr": {
    "triggers": [
      { "label": "automerge" },
      { "approved_by": ["alice", "bob"], "min_approvals": 2 }
    ]
  }
}

auto_label_pr — labels, auto-update, and auto-merge

A map of label name to settings:

FieldMeaning
auto_add"on_pr_creation" applies the label when a PR opens; false (default) disables.
create_label_if_missing_in_repotrue creates the label in the repo (using color) if absent.
color6-digit hex, with or without a leading # (default 00FF00).
mode"auto_update" keeps the branch in sync while the label is present. "auto_merge" bidirectionally syncs the label with GitHub's native auto-merge.
auto_merge_method"merge", "squash" (default), or "rebase" — only with mode: "auto_merge".

With mode: "auto_merge", adding the label arms native auto-merge and removing it disarms it; toggling auto-merge from the GitHub UI adds/removes the label to match. If the PR is already mergeable (GitHub won't arm auto-merge on a clean PR), pr-minder merges it directly with auto_merge_method. Branch protection still gates every merge.

{
  "auto_label_pr": {
    "automerge": { "mode": "auto_merge", "auto_merge_method": "squash", "create_label_if_missing_in_repo": true, "color": "0e8a16" },
    "auto-update": { "mode": "auto_update", "auto_add": "on_pr_creation", "create_label_if_missing_in_repo": true }
  }
}

auto_trigger_workflows — revive “zombie” PRs

A boolean (default false). A PR created with a workflow's default GITHUB_TOKEN (author github-actions[bot]) never triggers its own workflows — GitHub suppresses those runs to prevent recursion — so it sits open with no CI. pr-minder revives it by closing and reopening it with the App's credentials, firing a fresh pull_request.reopened event that runs the workflows.

  • On opened it revives PRs authored by github-actions[bot].
  • On reopened it revives any open PR that still has zero workflow runs, whatever its author.

It can't loop: pr-minder's own reopen returns as a bot-sent reopened event, which is skipped.

{ "auto_trigger_workflows": true }

auto_open_pr — open PRs for forgotten branches

Opens a PR for any branch that is ahead of its base and has no open PR, using pr-minder's App credentials so the PR triggers its workflows normally (never a zombie). This is the root-cause fix for “a CI job opened a PR with GITHUB_TOKEN and its checks never ran.”

FieldMeaning
enabledTurn the feature on (default false).
skip_branchesExtra branches to never open PRs for. The default branch and gh-pages are always skipped.
target_baseBase branch for opened PRs (default: the repo's default branch).

It runs on a push to a non-default branch and sweeps existing branches once when the App is installed or repositories are added. Fork branches are not opened.

{
  "auto_open_pr": {
    "enabled": true,
    "skip_branches": ["staging"],
    "target_base": ""
  }
}

Org-wide config with per-repo overrides

In the <org>/.github repo at .github/config/pr-minder/pr-minder.jsonc:

{
  "$schema": "https://raw.githubusercontent.com/wow-look-at-my/pr-minder/master/schema/pr-minder.schema.json",
  "auto_update_pr": { "triggers": [{ "label": "automerge" }] },
  "auto_trigger_workflows": true,
  "repos": {
    "special-repo": { "auto_update_pr": { "triggers": [{ "label": "ready-to-merge" }] } },
    "opt-out-repo": { "auto_update_pr": { "triggers": [] } }
  }
}

In overrides, auto_update_pr replaces the parent's entirely; auto_label_pr and auto_open_pr merge field-by-field.
